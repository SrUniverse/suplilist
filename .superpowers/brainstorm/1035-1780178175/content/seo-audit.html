🔴 Crítico — afeta indexação e analytics agora
1. URLs com hash: suplilist.com/#/home INDEXAÇÃO

app.js redireciona / → /home desnecessariamente. home-page.js usa window.location.hash nos botões em vez de pushState.
Google vê suplilist.com/#/home como fragmento, não como URL indexável.

2. Plausible não rastreia navegação interna ANALYTICS

Em SPAs, plausible('pageview') precisa ser chamado a cada mudança de rota. O router atual não notifica o Plausible — então você só vê a entrada inicial, nunca as páginas internas. O script instalado também usa URL customizada sem data-domain.

3. Links hash residuais: faq-page.js · checkin-page.js NAVEGAÇÃO

6 links internos ainda usam data-href="#/rota" ou href="#/my-stack". Quebram silenciosamente com History API — levam para âncora vazia.

🟡 Médio — ganhos reais de visibilidade
4. FAQPage schema JSON-LD ausente RICH SNIPPETS

O FAQ tem 11 perguntas e respostas de qualidade. Adicionando FAQPage schema em index.html (estático, visível para crawlers), o Google pode exibir as perguntas diretamente no resultado de busca — aumenta CTR sem nenhum esforço de link building.

5. Title dinâmico por página UX + SEO

Todas as páginas compartilham o mesmo <title>. Quando o usuário abre 3 abas (Catálogo, Calculadora, FAQ), todas mostram "SupliList | Suplementos...". Atualizar o title em cada rota melhora UX e ajuda o Google a entender cada página individualmente.

6. <noscript> ausente RASTREAMENTO

Bing e outros crawlers que não executam JS veem um <body> quase vazio. Um <noscript> com conteúdo resumido (nome do app, proposta de valor, links principais) garante alguma visibilidade nesses crawlers.

7. Preload de fontes ausente CORE WEB VITALS

Syne e Inter são carregadas via Google Fonts (render-blocking). Adicionar <link rel="preload"> para os arquivos woff2 críticos reduz o Largest Contentful Paint (LCP), que é fator de rankeamento.
